# Bronze to Silver ETL Process

## Overview

The **Bronze to Silver** pipeline transforms raw datasets stored in the **Bronze layer** into cleaned and validated datasets in the **Silver layer**.

At this stage, the pipeline performs several important tasks:

- Data cleaning
- Data validation
- Data type normalization
- Partition column generation
- Separation of invalid records

The main goal is to ensure that downstream processes only consume **high-quality and reliable datasets**.

The transformation logic is implemented in:

```
distribute/job/bronze_to_silver.py
```

---

# Input Data (Bronze Layer)

The pipeline reads raw datasets from the **Bronze storage layer**.

Location: 

```
distribute/data/bronze
```

Bronze Structure:

```
bronze
├── events
└── reference
```

### Events Dataset

The **events dataset** contains user activity logs generated by source systems.

| Column | Description |
|------|-------------|
| event_id | Unique identifier of the event |
| user_id | Identifier of the user |
| event_type | Type of user activity |
| event_ts | Timestamp when the event occurred |

---

# Transformation Process

The ETL job applies several transformations before writing the data into the Silver layer.

## 1. Data Cleaning

Basic cleaning operations are applied to ensure consistent values.

- Remove leading and trailing spaces
- Normalize text format
- Standardize event type values

Process Transformation:

```python
event_type = trim(event_type)
event_type = upper(event_type)
```

---

## 2. Data Type Normalization

Columns are converted to appropriate data types.

| Column | Target Type |
|------|-------------|
| event_ts | timestamp |
| user_id | string |

This ensures compatibility for downstream analytics and aggregation processes.

---

## 3. Partition Column Generation

A partition column is created from the event timestamp.

```
event_date = to_date(event_ts)
```

This allows efficient querying and storage optimization.

Partition Structure:

```
silver/clean/events/event_date=YYYY-MM-DD
```

---

# Data Validation

Records are validated before being written into the Silver layer.

An **event record is considered valid if:**

- `event_id` is not null
- `user_id` is not null
- `event_ts` is a valid timestamp

Records that fail validation will not be included in the clean dataset.

---

# Rejected Records Handling

Invalid records are separated and stored in a **rejected dataset**.

Common rejection cases:

- Missing `event_id`
- Missing `user_id`
- Invalid timestamp values

Rejected records are stored for:

- Data quality monitoring
- Debugging
- Pipeline validation

---

# Output Data (Silver Layer)

The processed datasets are written to the **Silver storage layer**.

Location:

```
distribute/data/silver
```

Project structure:

```
silver
├── clean
│   ├── events
│   │   └── event_date=yyyy-mm-dd
│   │
│   └── users
│
└── rejected
    └── events
```

### Clean Dataset

Contains **validated and cleaned records** ready for downstream pipelines.

Path:

```
silver/clean/events/event_date=YYYY-MM-DD
```

### Rejected Dataset

Contains invalid records that failed validation checks.

Path:

```
silver/rejected/events
```

---

# Running the Bronze to Silver Pipeline

To execute the **Bronze → Silver ETL process**, run the following command:

```powershell 
spark-submit distribute/job/bronze_to_silver.py --config distribute/config/config.yaml
```

The configuration file defines:

- Bronze input locations
- Silver output locations
- Pipeline settings

Configuration file location:

```
distribute/config/config.yaml
```

---

# Integration with Airflow

This ETL step is orchestrated by the Airflow DAG located at:

```
airflow/pipeline_dag.py
```

The DAG schedules and executes the pipeline stages in sequence:

1. Raw → Bronze  
2. **Bronze → Silver**  
3. Silver → Gold
